# Feature Specification: Chatbot Memory System

**Feature Branch**: `008-chatbot-memory-system`
**Created**: 2026-03-15
**Status**: Draft
**Input**: User description: "Mnemo — 4-phase chatbot memory system for teaching LLM memory architecture"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 基础对话与消息持久化 (Priority: P1)

作为一个用户，我可以与 AI 聊天助手进行多轮对话，对话历史在页面刷新后依然保留，并且我可以管理多个独立的会话。

**Why this priority**: 这是整个应用的基础——没有持久化的对话和会话管理，后续的摘要、记忆、RAG 都无从谈起。这是最小可用产品 (MVP)。

**Independent Test**: 打开应用，创建一个新会话，发送几条消息并收到流式回复，刷新页面后消息仍在，切换到另一个会话并确认历史独立。

**Acceptance Scenarios**:

1. **Given** 用户打开应用, **When** 发送一条消息, **Then** AI 以流式方式逐字回复，消息气泡区分用户和 AI，并显示时间戳
2. **Given** 用户在一个会话中发送了多条消息, **When** 刷新页面, **Then** 所有历史消息从数据库恢复，按时间顺序显示
3. **Given** 用户有多个会话, **When** 在侧边栏点击另一个会话, **Then** 消息区切换到该会话的历史消息
4. **Given** 用户点击"新建会话"按钮, **When** 新会话创建成功, **Then** 消息区清空，侧边栏新增一个条目
5. **Given** 用户删除一个会话, **When** 删除操作完成, **Then** 该会话从列表中消失（软删除），数据库保留原始数据
6. **Given** 会话中已有超过 20 条消息, **When** 用户发送新消息, **Then** 系统只取最近 20 条消息作为上下文发送给 LLM
7. **Given** 最近消息的 token 总数超过预算（默认 8000 tokens）, **When** 发送请求, **Then** 从最旧的消息开始裁剪，确保不超预算
8. **Given** AI 回复包含 Markdown 格式（代码块、列表）, **When** 显示回复, **Then** Markdown 被正确渲染

---

### User Story 2 - 会话摘要与上下文压缩 (Priority: P2)

作为一个用户，当我与 AI 进行长时间对话时，系统自动将较旧的对话内容压缩为摘要，让 AI 既能记住之前聊过什么，又不会因为上下文窗口溢出而"失忆"。

**Why this priority**: 长对话是真实聊天场景的核心痛点。上下文窗口有限，不压缩就会丢失早期重要信息。此功能直接解决"聊着聊着 AI 就忘了"的问题。

**Independent Test**: 在一个会话中发送超过 20 条消息，确认系统生成了摘要，后续 AI 回复仍能引用早期讨论的内容，打开 Debug 面板确认摘要和 token 分布。

**Acceptance Scenarios**:

1. **Given** 会话中未被摘要覆盖的消息数达到 20 条, **When** 用户发送新消息, **Then** 系统异步生成/更新摘要，不阻塞当前回复
2. **Given** 摘要已生成, **When** 用户继续对话, **Then** 上下文由"System Prompt + 摘要 + 最近 10 条原始消息"组成
3. **Given** 已有旧摘要且又累积了 20 条新消息, **When** 触发摘要, **Then** 旧摘要与新消息一起重新总结，生成递进式更新摘要
4. **Given** 用户展开 Debug 面板, **When** 查看面板内容, **Then** 可见当前摘要文本、本次请求的总 token 数、各部分占比、最近摘要更新时间

---

### User Story 3 - 长期记忆系统 (Priority: P3)

作为一个用户，AI 能在不同会话之间"记住"我透露的个人信息、偏好和事实，让每次新对话都不用从头自我介绍。

**Why this priority**: 跨会话记忆是 Chatbot 从"工具"升级为"助手"的关键能力。但它依赖 Phase 1 的基础对话和 Phase 2 的上下文管理能力。

**Independent Test**: 在会话 A 中告诉 AI"我是一名后端工程师，喜欢用 Go"，关闭会话 A，新建会话 B 问"你知道我的技术栈吗"，AI 能准确回答。在记忆管理页面确认该条记忆存在并可编辑。

**Acceptance Scenarios**:

1. **Given** 用户在对话中提到个人信息（如"我住在上海"）, **When** 触发记忆提取, **Then** 系统自动提取并存储该事实为一条简洁的记忆条目，带有分类标签
2. **Given** 用户在新会话中发送消息, **When** 系统检索到相关记忆, **Then** 相关记忆注入到 System Prompt 中，AI 回复体现出对用户的了解
3. **Given** 用户打开记忆管理页面, **When** 页面加载, **Then** 列出所有记忆条目（内容、分类、创建时间、最后命中时间），支持编辑、删除和手动添加
4. **Given** 对话已进行 5 轮, **When** 系统执行批量记忆提取, **Then** 从最近消息中提取新记忆，且不阻塞对话
5. **Given** 用户消息包含关键词"记住"（如"记住我不吃辣"）, **When** 消息发送, **Then** 立即触发记忆提取
6. **Given** 新提取的记忆与已有记忆语义相似度 > 0.85, **When** 保存记忆, **Then** 更新已有记忆而非创建重复条目
7. **Given** Debug 面板已打开, **When** 查看记忆检索结果, **Then** 可见检索到的记忆条目及其相似度分数

---

### User Story 4 - RAG 知识检索 (Priority: P4)

作为一个用户，我可以上传文档（.txt/.md），AI 能基于这些文档内容回答我的问题，并标注引用来源，同时与记忆系统协同工作。

**Why this priority**: RAG 是记忆系统的最后一层（语义记忆），为 Chatbot 赋予特定领域知识。它依赖前三个 Phase 的基础设施。

**Independent Test**: 上传一份产品文档，向 AI 提问文档中的具体细节，AI 能准确回答并标注来源文件名。Debug 面板显示检索到的文档块和 token 预算分配。

**Acceptance Scenarios**:

1. **Given** 用户上传一个 .md 文件, **When** 上传完成, **Then** 系统自动分块处理（300-500 tokens/块，50 tokens 重叠），生成 embedding 并存入数据库，前端显示处理状态
2. **Given** 知识库中已有文档, **When** 用户提问与文档内容相关的问题, **Then** 系统检索最相关的文档块，注入上下文，AI 回复附带来源标注
3. **Given** 用户既有长期记忆又有知识库文档, **When** 用户提问, **Then** 记忆搜索和文档搜索并行执行，两类结果分别注入上下文的指定位置
4. **Given** 完整上下文组装完成, **When** 发送给 LLM, **Then** 上下文按以下顺序：System Prompt → 长期记忆 → RAG 结果 → 会话摘要 → 对话历史 → 当前消息，各部分有独立 token 预算
5. **Given** Debug 面板已打开, **When** 查看本次请求信息, **Then** 可见检索到的记忆条目（含分数）、文档块（含来源和分数）、完整 token 预算分配

---

### Edge Cases

- 当会话中只有 1 条消息时，上下文管理不应触发摘要或裁剪
- 用户连续快速发送多条消息时，消息持久化应按序处理，不丢消息
- 摘要生成过程中用户继续发消息，新消息不应被遗漏
- 记忆提取失败时，正常对话不受影响，错误静默记录
- 上传空文档或极小文档（不足一个 chunk）时，应给出合理提示
- LLM 调用失败时，显示友好错误消息并支持重试
- 数据库连接失败时，聊天功能降级为无记忆模式（仅使用当前输入）
- 记忆检索返回相似度全部低于阈值 (0.3) 时，不注入任何记忆

## Clarifications

### Session 2026-03-15

- Q: 会话标题如何生成？ → A: 首条消息发送后，由 LLM 自动生成标题
- Q: 上传的文档是否支持删除？ → A: 支持删除，同时级联删除关联的 chunks 和 embeddings

## Requirements *(mandatory)*

### Functional Requirements

**Phase 1 — 基础对话**
- **FR-001**: 系统 MUST 提供标准聊天界面（消息列表、输入框、发送按钮），支持 Markdown 渲染
- **FR-002**: 系统 MUST 支持流式输出，逐字显示 AI 回复
- **FR-003**: 系统 MUST 在侧边栏展示会话列表（按最近更新排序），支持新建、切换、删除（软删除）会话。新会话在首条消息发送后由 LLM 自动生成标题
- **FR-004**: 系统 MUST 将每轮对话实时存入数据库，页面刷新后完整恢复
- **FR-005**: 系统 MUST 实现滑动窗口上下文管理——最多取最近 20 条消息，且总 token 不超过可配置预算（默认 8000）

**Phase 2 — 会话摘要**
- **FR-006**: 系统 MUST 在未摘要消息数达到阈值（默认 20 条）时自动异步生成摘要
- **FR-007**: 系统 MUST 将摘要作为 system 消息注入上下文，与最近 N 条原始消息组合
- **FR-008**: 系统 MUST 支持递进式摘要——旧摘要与新消息合并重新总结
- **FR-009**: 系统 MUST 提供可折叠的 Debug 面板，显示摘要内容、token 分布、最近更新时间

**Phase 3 — 长期记忆**
- **FR-010**: 系统 MUST 能自动从对话中提取用户事实，分类为 preference/fact/behavior
- **FR-011**: 系统 MUST 在用户发消息时基于语义相似度检索 Top-K（默认 5）条最相关记忆并注入 System Prompt
- **FR-012**: 系统 MUST 提供记忆管理页面，支持查看、编辑、删除和手动添加记忆
- **FR-013**: 系统 MUST 实现智能触发策略——每 5 轮批量提取、关键词即时触发、会话结束终结提取
- **FR-014**: 系统 MUST 对相似度 > 0.85 的记忆执行更新而非新增（去重）

**Phase 4 — RAG 知识检索**
- **FR-015**: 系统 MUST 支持上传 .txt 和 .md 文档，自动分块（300-500 tokens/块，50 tokens 重叠）、生成 embedding 并存储。支持删除文档，级联删除关联的 chunks 和 embeddings
- **FR-016**: 系统 MUST 在用户提问时并行检索知识库和长期记忆，结果分别注入上下文指定位置
- **FR-017**: 系统 MUST 实现完整的分层上下文组装（System Prompt → 记忆 → RAG → 摘要 → 历史 → 当前消息），各部分有独立 token 预算
- **FR-018**: Debug 面板 MUST 增强展示：检索到的记忆和文档块（含相似度分数）、token 预算分配可视化

**跨 Phase 需求**
- **FR-019**: 摘要生成、记忆提取等重计算 MUST 异步执行，不阻塞用户实时聊天
- **FR-020**: 所有 token 预算和触发阈值 MUST 集中配置、可调整

### Key Entities

- **Conversation（会话）**: 用户的一个对话上下文，包含标题、创建/更新时间
- **Message（消息）**: 单条对话消息，属于一个会话，区分角色（用户/AI/系统），记录 token 数
- **Summary（摘要）**: 会话级别的对话摘要，记录覆盖的消息数和 token 数
- **Memory（记忆）**: 跨会话的用户事实条目，带分类标签、向量 embedding、访问统计
- **Document（文档）**: 用户上传的知识库文档，记录文件名和分块数
- **DocumentChunk（文档块）**: 文档的分块单元，带向量 embedding，用于语义检索

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户发送消息后，首个 AI 回复 token 在 1.5 秒内开始流出 (TTFT < 1.5s)
- **SC-002**: 上下文组装（含摘要注入、记忆检索、RAG 检索）总耗时 < 300ms
- **SC-003**: 页面刷新后，100% 的历史消息从数据库完整恢复，无数据丢失
- **SC-004**: 长对话（50+ 条消息）场景中，AI 仍能引用早期讨论的关键事实（通过摘要保留）
- **SC-005**: 跨会话记忆准确率——在会话 A 告知的个人信息，在会话 B 中至少 80% 能被 AI 正确回忆
- **SC-006**: 上传文档后，针对文档内容的提问，AI 能在 90% 的情况下给出包含正确来源标注的回答
- **SC-007**: Debug 面板能准确显示每次请求的完整上下文构成，帮助学习者理解系统内部运作
- **SC-008**: 记忆提取、摘要生成等后台操作不导致用户可感知的对话延迟（操作完全异步）

### Assumptions

- 使用硬编码的 `userId` 做演示，不实现用户认证系统
- UI 以功能演示为目标，不追求生产级视觉设计
- 单用户场景，不考虑多用户并发写入记忆的冲突
- Token 计算使用简易估算（英文 字符数/4，中文 字符数/1.5），不需要精确 tokenizer
- 各 Phase 的默认参数值（token 预算、阈值等）基于教学场景合理推测，可通过配置调整
