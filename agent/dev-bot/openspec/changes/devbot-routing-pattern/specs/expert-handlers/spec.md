## ADDED Requirements

### Requirement: Unified Handler interface
All expert handlers SHALL implement a common `Handler` interface with a `name` property and a `handle(input: string, context?: ConversationContext): Promise<string>` method. Each handler SHALL have its own dedicated System Prompt and LLM configuration.

#### Scenario: Handler returns formatted response
- **WHEN** any handler's `handle` method is called with valid input
- **THEN** it returns a string response generated by the LLM using the handler's dedicated System Prompt

### Requirement: Code Explainer handler
The Code Explainer handler SHALL accept code snippets and user questions, then produce structured explanations in Markdown format following the structure: Overview → Section-by-section explanation → Key takeaways.

#### Scenario: Explain a code snippet
- **WHEN** Code Explainer receives input containing a code block and a question about its logic
- **THEN** it returns a structured Markdown explanation with overview, detailed walkthrough, and key points

### Requirement: Bug Detective handler
The Bug Detective handler SHALL analyze error messages and problematic code, producing structured diagnostic output following: Symptom identification → Root cause analysis → Fix suggestion → Prevention advice.

#### Scenario: Diagnose a TypeError
- **WHEN** Bug Detective receives "TypeError: Cannot read properties of undefined (reading 'map')"
- **THEN** it returns analysis with symptom, likely causes ranked by probability, concrete fix, and prevention tips

### Requirement: Code Generator handler
The Code Generator handler SHALL accept natural language requirements and generate code. It SHALL use a slightly higher temperature (0.3-0.5) than other handlers. Output SHALL include: requirement confirmation → generated code → usage instructions.

#### Scenario: Generate a function from description
- **WHEN** Code Generator receives "帮我写一个防抖函数"
- **THEN** it returns generated code with explanation and usage example

### Requirement: Doc Searcher handler
The Doc Searcher handler SHALL answer questions about APIs, frameworks, and libraries. It SHALL simulate a RAG-like flow by using LLM knowledge to provide answers with source references. Output SHALL include the answer and documentation source citations.

#### Scenario: Answer a framework usage question
- **WHEN** Doc Searcher receives "useEffect 的依赖数组怎么用？"
- **THEN** it returns an answer with relevant documentation references

### Requirement: Clarifier handler
The Clarifier handler SHALL NOT directly answer the user's question. Instead, it SHALL generate 1-2 targeted clarifying questions to help determine the user's actual intent. The clarified response SHALL be re-routed through the routing pipeline.

#### Scenario: Generate clarifying questions for vague input
- **WHEN** Clarifier receives "这个东西能不能优化一下"
- **THEN** it returns 1-2 specific clarifying questions asking what type of optimization the user wants

#### Scenario: Clarified input re-enters routing
- **WHEN** user responds to clarifying questions with more specific input
- **THEN** the response is sent back through the routing pipeline with full conversation context
